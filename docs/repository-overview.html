1. プロジェクト概要

このリポジトリは、高校生・高専生対象のプログラミング大会 「Supercomputing Contest 2026 (SuperCon)」本選 推測(対象・大会名は文脈/外部情報) で、スーパーコンピュータ「富岳」(CPU: A64FX / ARM + SVE 512bit) 確認済(公式サイトで富岳使用を確認) を使って即座に開発を始めるためのテンプレート集 + 開発支援ツール群です。

中身は大きく3層に分かれます。(A) 富岳向けに最適化した MPI+OpenMP の C++ テンプレート、 (B) 手元PC(WSL)から富岳へ「転送→ビルド→ジョブ投入→結果回収」を行うリモート開発ツール、 (C) パラメータを自動で振って速度を測る自動チューニング測定基盤。

規模

約2,500行
C++/シェル/Python のソース合計 (wc -l 概算)

主要言語

C++17 / Bash / Py
+ Make, PJM ジョブ, rsync/ssh

対象機

富岳 A64FX
1ノード=48コア=4CMG×12

外部依存

ほぼゼロ
CDN/外部ライブラリ不使用

2. 何を解決するリポジトリか

競技本選では「正しく動くプログラムを、富岳上でできるだけ速くする」のが目標です。そこで生じる 3つの困りごとを、このリポジトリが先回りして解決します。

困りごとこのリポジトリの解決策
当日ゼロから富岳向け MPI+OpenMP コードを書く時間がない 課題タイプ別の最適化済みテンプレート(探索/ステンシル/汎用)を用意。当日は中身だけ差し替える
富岳に AI(Claude等)を持ち込めない。状態が手元で見えない 富岳での実行結果を results/<jobid>/ に「完全スナップショット」として落とし、手元のAIがファイルを読むだけで状況把握できるようにする
「パラメータを振って速さを測る」作業が退屈で高コスト 1ジョブでN構成をまとめて測る測定ハーネスで自動化し、人間の脳をアルゴリズム設計に集中させる
設計思想は docs/design-rebuild.md(C++テンプレート)と docs/design-infra.md(インフラ/ハーネス)に「なぜそう作ったか」がまとまっています。確認済

3. アーキテクチャ全体像

「AIや解析は手元(WSL)、富岳は計算するだけ」という責務分離が中核です。 富岳には API キーや余計な依存を持ち込まず、コンパイル済み C++ を走らせて測るだけのバックエンドに徹します。

┌──────────── 手元 (WSL2 + AI/Claude) ────────────┐      ┌──── 富岳 (A64FX) ────┐
│  src/ 編集 ・ ツール実行 ・ 結果解析 ・ 状態管理   │      │  ソース受領          │
│                                                  │ ssh  │  mpiFCCpx ビルド     │
│  tools/fugaku-run.sh ─ sync→submit→wait→fetch ──┼─────▶│  pjsub ジョブ実行    │
│                                                  │rsync │  結果を results/ へ  │
│  results// を AI が読む ◀─────────────────┼──────┤  (出力/資源/状態)    │
└──────────────────────────────────────────────────┘      └──────────────────────┘

自動で回り続ける「ループ」ではなく、人間が1コマンド叩くたびに1回動く手動フローです。 （普段のローカル開発で「実行して出力を見る」のと同じ感覚を、富岳に対して再現するのが狙い）

5. 主要ファイル解説

C++ テンプレート (src/)

ファイル役割行数
common.hpp共通ヘッダ。高分解能タイマー wtime() と乱数 Rng(xoshiro256**)52
utilities.hpp競技用ユーティリティ。超高速I/O fastio、時間予算 Budget、ビット演算187
tune_args.hpp全テンプレ共通のチューニング規約。argv/env を読み、末尾に #TUNE elapsed=.. score=.. correct=.. を出す105
skeleton.cpp汎用スケルトン(N体/線形代数/モンテカルロ向け)。MPI+OpenMP+時間予算ループ103
stencil.cpp2D格子シミュレーション(反応拡散/CA等)。行分割ハロ交換 + 通信計算オーバーラップ170
stencil_blocked.cpp温度ブロッキング版(メモリ律速時の高速化)。タイル寸法 BT/RB/CB はコンパイル時定数231
search.cpp並列焼きなまし(SA)。組合せ最適化向け。スレッド独立試行 + MPI_MAXLOC で全体ベスト共有179
contest.cpp当日雛形。「読む→解く→書く」I/O配線の実証(合計と最大値を返すだけのサンプル)63
solver_naive.cpp愚直解プレースホルダ(ストレステスト用、現状ほぼ空)14

富岳リモート操作 (tools/) — 単発実行

ファイル役割
fugaku-run.shワンショット実行。下記4本を順に呼び、最後に状態サマリを表示
fugaku-sync.shsrc/→REMOTE/src/ と Makefile を rsync、ログインノードで make fugaku(ビルドログを build.log に保存)
fugaku-submit.shPJM ジョブを動的生成して pjsub。git来歴・資源計測・status をジョブに埋める
fugaku-wait.shpjstat をポーリングして完了待ち、最終状態を記録
fugaku-fetch.sh結果を回収し、全部入りの meta.json(設定+来歴+ビルド+資源+結末)を生成
fugaku-cancel.shpjdel ラッパ。暴走・ミスジョブを即停止

富岳環境ノブ(FUGAKU_CXX=mpiFCCpx / module load / LLIO volume / freq / --spath)は tools/fugaku-config.env 駆動の opt-in(既定空=現状維持)。詳細は docs/fugaku-workflow.md。

6. データ/処理フロー

(1) 通常の開発フロー(単発)

src/ 編集→ make / make test-mpi (ローカル健全性)→ fugaku-run.sh→ results/<jobid>/ 生成→ AI が読む → 修正

各ジョブの results/<jobid>/ は「そのジョブの検死報告書」として自己完結します:

ファイル意味
meta.json全部入りヘッダ(設定・git commit・ビルド状態・終了コード・実時間・最大メモリ・結末)
build.logmpiFCCpx ビルド出力(富士通clang のエラー/警告)
stdout.txt / stderr.txtプログラム出力 / 実行時エラー
resource.txt/usr/bin/time -v による最大RSS・実時間・CPU%(存在する場合のみ)
status.txt正常完了の印(無ければスケジューラに kill された未完を示す)

(2) 自動チューニングフロー

autotune/spaces/<tmpl>.tsv
        │  gen_configs.py (LHS/grid)
        ▼
   configs.tsv ──fugaku-tune.sh──▶ 富岳で1ジョブにN構成を掃引 (tune-sweep.sh)
                                          │
                                          ▼
                                    results.csv ──update_incumbent.py──▶ state/incumbent.json
                                                                          (= いつ落ちても提出できる現時点ベスト)

configs.tsv ↔ results.csv ↔ incumbent.json という固定インターフェースが本体で、 「configsを作る側」(手動 / LHS / 将来のGP)は差し替え可能なプラグインとして設計されています。

7. ビルド・テスト・実行コマンド

以下は Makefile と各スクリプトで確認したコマンドです。make test-mpi はこの環境で実行し全PASSを確認済 確認済。 富岳向けコマンド(make fugaku / fugaku-*.sh)は実機が必要なため未実行 推測:実機Day1で初検証。

ローカル(手元PC)

# 全テンプレートをビルド (g++ / OpenMP / MPIなし)
make                 # build/{skeleton,stencil,stencil_blocked,search}
make contest         # 当日雛形をビルド

# MPI 経路をローカル4ランクで自動検証 (富岳に投げる前の健全性チェック)
make local-mpi       # mpic++ で build/mpi/* を生成 (要 OpenMPI)
make test-mpi        # ハロ交換/Allreduce/MAXLOC を自動 PASS/FAIL 判定

# テストケース一括判定
bash tests/judge.sh tests

# 掃引のローカル予行演習
tools/gen_configs.py autotune/spaces/search.tsv --n 6 --ranks 2 --omp 2 --rep 1 > /tmp/c.tsv
tools/tune-local.sh /tmp/c.tsv 0.5 max-score

富岳(本選環境)

# 富岳提出用ビルド (富士通クロスコンパイラ mpiFCCpx)
make fugaku BUDGET_SEC=1750     # 1750 は仮の制限値。当日の実時間制限で上書き

# 単発実行 (転送→ビルド→投入→待機→回収)
tools/fugaku-run.sh stencil 1750

# 1ジョブで複数構成を掃引
tools/fugaku-tune.sh configs.tsv 1750 max-score

使用した読み取り確認コマンド: find, wc -l, git log --oneline, ls, command -v cargo, bash -n, make test-mpi。

8. 技術的な特徴

A64FX 最適化

1ランク=1CMG固定、first-touch、SVE512ベクトル化、stride パディング、-Kzfill、通信計算オーバーラップ

ローカル検証可能

富岳に投げる前に make test-mpi で MPI バグ(ハロ境界/集約漏れ/デッドロック)を潰せる

中断耐性

1構成ごと即追記 + sync。途中で kill されても測れた分と incumbent が残る(anytime設計)

完全な状態可視化

富岳ジョブを1ディレクトリの自己完結スナップショットに。AIがローカルを読むだけで把握

依存ゼロ志向

bash + Python標準ライブラリのみ。富岳側に Python 依存を置かない

堅牢性

敵対的テストで多数の入力異常(CRLF/不正値/crash偽陽性等)を修正済

Codex セカンドオピニオン

Codex CLI(OpenAI)を read-only で独立レビュー/検証に利用。Claude と同一情報を読み、トークンを分散
main には PR #1〜#6 がマージ済み(autotuneハーネス/snapshot、富岳コマンド準拠強化、Codex導入、C++バグ修正、ハーネス堅牢化、HTML総覧)。確認済(git log)
