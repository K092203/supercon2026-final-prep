## 1. 核心思想

本選を決めるのは**課題のアルゴリズム**。最大化すべきは:

```
最終性能 / (消費した富岳ジョブ予算 + 消費した人間の思考時間)
```

- **桁を動かすのはアルゴリズム(10〜100×)であり、パラメータ調整(1.5〜3×)ではない。**
  ゆえに自動化は軽量に留め、浮いた脳をアルゴリズム設計へ回す。
- ジョブ枠(キュー待ち+ノード確保)と 3 人の脳は**最も希少な資源**。
  「パラメータを振って速さを測る」退屈で高価な部分だけを切り出し、希少資源を最終性能に変えるレートを上げる。

この方針が、以下すべての設計判断の上位前提。

---

## 2. 責務分離: AI を富岳に入れない

```
WSL2 側 (頭脳)                      富岳 (測定器)
  AI / Claude Code                    ソースを受け取る
  configs 生成 (LHS / 将来GP)         mpiFCC ビルド
  results 解析・incumbent 管理         ジョブ実行 = N構成の測定
  ソース編集・state 永続化             results.csv 出力
```

**なぜ**: API key・`node_modules`・PyTorch 等を富岳に持ち込まない。富岳は「コンパイル済み C++ を
走らせて測る」純粋なバックエンドに徹する。これにより富岳ログインノードでの作業を最小化し、
オフライン・依存ゼロの運用が成立する。AI・GP・解析・state は全て手元(WSL)に置く。

---

## 3. WSL↔富岳 開発ループの設計

1 ジョブ = **「ノードを 1 回確保している間に最大の情報を回収する単位」**。高い買い物(キュー待ち)を
1 回だけ払い、その中で評価を稼ぐ。

- **単発デバッグ**: `fugaku-run.sh`(sync→submit→wait→fetch)。1 構成を正しく速く測る。
- **バッチ掃引**: `fugaku-tune.sh`。1 ジョブで N 構成を測り切る(§5)。
- **ControlMaster/OTP**: 初回 ssh で OTP 入力 → `ControlPersist` 中は再認証不要 → スクリプトが完全自動化。
- **anytime**: ジョブが時間切れで殺されても、測れた分は必ず残す(後述の即追記)。
- **1ジョブ = 完全スナップショット(検死報告書)**: `results/<jobid>/` に config だけでなく
  build.log(どう作ったか)/ commit(どのコードか)/ resource.txt(どう走ったか)/ status(どう終わったか)を
  全て落とす。富岳に AI を入れない代わりに、**AI がローカルファイルを読むだけで富岳の状態を完全把握**できる
  ようにするのがこのインフラの核心。失敗理由・遅さ・kill 有無が meta.json 1 つで分かる。

## 4. オートチューニング基盤の設計

### 4.1 インターフェースが製品

```
configs.tsv  ↔  results.csv  ↔  incumbent.json
```

この 3 つの固定インターフェースが本体。**configs を作る側**(手動 / LHS / 将来の GP)は
すべてこの裏の差し替え可能なプラグイン。これにより「GP を一切書かなくても完成品」になり、
後から GP を足しても本体を壊さない。

### 4.2 測定基盤が本体・GP は「外せる頭」

GP(trustbo-gp)を主役に据えない。理由は §1: 桁を動かすのはアルゴリズムで、GP が詰めるのは
1.5〜3× の部分。少ラウンド(4 日・有限ジョブ枠)では GP の標本効率の優位はほとんど出ない。
よって **測定ハーネス(高価値・低リスク)を本体に、GP は任意の上積み**とし、失敗時は劣化継続する:

```
GP ask_batch  →(失敗)→  LHS / random  →(失敗)→  手動 baseline
```

### 4.3 測定ループは 1 本・launcher だけ差し替え

`tune-sweep.sh` が唯一の測定ループ。ローカルは `mpirun`/`build/mpi`、富岳は `mpiexec`/`build/fugaku` と
**launcher と bindir だけ**が違う。**なぜ**: 本選前は実機が無い。同一スクリプトをローカルで回せば、
富岳本番経路をそのまま予行演習でき、経路の乖離が生じない(DRY)。

### 4.4 探索空間は 3 層に割る

| 層 | 例 | 振り方 | なぜ分けるか |
|---|---|---|---|
| リビルド要 | flagset, `-D`定数(BT/RB/CB), stride | 事前ビルドし `bin` 列で選択 | 実行時に変えられない |
| ジオメトリ(ピン留め敏感) | ranks×omp | 別ジョブで少数点 + pin 検証 | 1ジョブ内で振ると mis-pin が測定を汚染 |
| 連続な内側ノブ | sa_temp, cooling, block, pad | 1ジョブ内バッチ掃引 | LHS/GP の本来の獲物 |

カテゴリ(排他的選択)は GP に渡さず**外側で列挙**。relax+round は GP に不自然なため。

### 4.5 anytime + incumbent = 命綱

- **1 構成測るたびに results.csv へ即追記 + sync**。時間切れで殺されても全滅しない。
- **incumbent.json** = 「現時点ベストの正解構成」。いつジョブが落ちても即提出できる状態を切らさない。
  厳密に改善した時だけアトミックに更新(途中で落ちても壊れない)。

### 4.6 目的関数の扱い

- 速度律速(ステンシル等) → `min-elapsed` / 予算固定の anytime(SA・MC) → `max-score`。
- **不正解・timeout は巨大値でなく有限ペナルティ**(`2×budget`)。GP のスケールを壊さないため。
- (GP 接続時)実行時間は `log` 変換、noise 下限を上げる。共有スパコンの計測ばらつきへの過学習を防ぐ。

## 5. 技術選定と不採用

| 判断 | 採用 | 理由 / 不採用の理由 |
|---|---|---|
| ハーネスのグルー言語 | **bash + Python(標準ライブラリのみ)** | 依存ゼロ・どこでも動く。Python は WSL 側のみ(富岳に Python 依存を置かない) |
| 測定ループの配置 | **bash 1 本(富岳へ rsync)** | 富岳側に Python を要求しない。awk で中央値集計 |
| configs 形式 | **TSV** | bash/awk/Python のどれでも壊れにくく扱える |
| 構造化出力 | **stderr の `#TUNE` 行** | stdout(=課題の解)を汚さない |
| GP エンジン | **trustbo-gp(任意・後付け)** | pure Rust・依存軽・オフライン vendor 可。ただし非クリティカル |
| Optuna | 不採用 | pip 前提。本選環境での配備が不確実 |
| BoTorch/Ax | 不採用 | PyTorch on ARM の配備が事故要因。低次元に過剰 |

判断軸は理論性能ではなく**「4 日・富岳・オフライン・時間が命」という運用環境**。「確実に動く・
自分で直せる」が勝つ。

## 6. 堅牢性の設計原則

3 ラウンドの敵対的テスト(詳細な穴一覧は [autotune.md](autotune.md) §11)を通じて確立した原則:

- **クラッシュではなく劣化**: 1 構成が失敗しても per-config timeout でペナルティを記録して継続。GP 失敗は fallback。
- **ジョブ kill 耐性**: 1 構成 1 行の atomic append(<4KB)で、SIGTERM 中でも results.csv は完全行のみ。
- **整合性違反は fail-fast**: 重複 id など「取り違え」を生む入力は即中止(黙って誤らない)。
- **入力は正規化・サニタイズ**: CRLF 除去、末尾改行なし対応、非数値 ranks/omp/rep は警告して既定値へ。
- **非ゼロ終了は信用しない**: crash/abort 後に `#TUNE` が出ていても失敗扱い(偽陽性を排除)。
- **再現可能性の保証**: configs と紐づかない測定値は incumbent にしない(提出不能を防ぐ)。

## 7. スコープ外(今回やらないこと)

- GP(trustbo-gp)の本接続・`cargo vendor`・合成ベンチでのストレステスト → 余力があれば後付け(差込点は autotune.md §9)。
- 富岳での実機初通し → Day1(本選前はローカル MPI で全経路を検証済み)。
- 本選レギュレーションの最終確認(AI/自作ツール/持ち込み/時間制限)→ Day1。
