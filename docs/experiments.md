# 最適化レシピ集 + 実験ログ

> 「速くしたい」と思ったら上から順に試す。各レシピは ROI(効果/手間) 順。
> 1 回 1 変更 → 計測 → 下の表に記録（複数同時に変えると原因が分からなくなる）。

---

## A. まず疑うこと（速度が出ない時の診断順）

1. **1ランク=1CMG になっているか** → `#PJM --mpi "max-proc-per-node=4"`（崩れると帯域半減）
2. **first-touch しているか** → 配列を使うスレッドが最初に書き込む並列初期化（stencil 実装済み）
3. **内側ループが SVE 化されているか** → unit-stride + `#pragma omp simd` + `__restrict` + 分岐レス
4. **メモリ律速か演算律速か** → 律速側を攻める（下記 B / C）

---

## B. メモリ律速（ステンシル等）のレシピ

| # | レシピ | 効果 | 手間 |
|---|---|---|---|
| B1 | リーディング次元を 2 のべきにしない（stride パディング。実装済み） | 中〜大 | 小 |
| B2 | `-Kzfill`（書き込み専用ストリームの read-for-ownership 省略。実装済み） | 中 | 小 |
| B3 | float 化（double で足りるか確認。SVE レーン 2倍・帯域半分） | 大 | 小〜中 |
| B4 | 通信/計算オーバーラップ（Irecv→内部計算→Waitall→境界。stencil 実装済み） | 中 | 中 |
| B5 | **温度ブロッキング**（stencil_blocked.cpp）: 複数ステップを L2 上で回し HBM 往復を削減 | 大 | 大 |
| B6 | ラージページ（`XOS_MMM_L_*`）で TLB ミス削減（submit にコメント記載） | 中 | 小 |
| B7 | 配列を 2 本 swap で使い回し、step ごとの malloc/zero を避ける（実装済み） | 中 | 小 |

A64FX 1ノードのメモリ帯域上限 ≈ 1 TB/s。達成帯域 = 転送バイト/秒 がこれに近ければメモリ律速で頭打ち
→ B5（演算あたりのメモリアクセスを減らす）でしか伸びない。

## C. 演算律速（探索・N体・密行列等）のレシピ

| # | レシピ | 効果 | 手間 |

## D. 計測の仕方

```bash
# ローカルで相対比較（A64FX 絶対値ではないが、変更の効果方向は分かる）
make stencil && time ./build/stencil
# 富岳実測
tools/fugaku-run.sh stencil <BUDGET_SEC>
cat results/latest/stdout.txt        # sum / steps / 経過秒
# スレッドスケーリング
OMP_NUM_THREADS=1  ./build/stencil
OMP_NUM_THREADS=12 ./build/stencil   # ~12x 近ければ並列効率良好
```
